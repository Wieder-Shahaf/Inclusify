## Summary

This phase trains a LoRA adapter on Qwen2.5-3B-Instruct-GPTQ-Int4 for LGBTQ+ inclusive language classification. The approach uses QLoRA (4-bit quantized LoRA training) on an English-only dataset (1001 samples) to validate the architecture before investing in Hebrew data collection. Research confirms this is a standard approach with well-established tooling.

**Primary recommendation:** Use Hugging Face PEFT + transformers + TRL SFTTrainer stack with QLoRA configuration. Run automated grid search across 9 configurations (3 ranks × 3 dropout values) to empirically identify optimal hyperparameters. Target validation accuracy >80% as success threshold before Hebrew expansion.

**Key insight:** GPTQ + QLoRA enables training a 3B parameter model on T4 16GB VRAM with batch_size=4, making this feasible on the existing Azure VM infrastructure.

### Pattern 1: QLoRA Configuration with GPTQ Base Model
**What:** Load GPTQ-quantized base model, attach LoRA adapters, train only adapters.
**When to use:** Training on consumer/T4 GPUs with limited VRAM.
**Example:**
```python
# Source: https://huggingface.co/Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import LoraConfig, get_peft_model

# Load GPTQ base model (already 4-bit quantized)
model = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4",
    torch_dtype="auto",  # Uses fp16 on T4
    device_map="auto"
)
tokenizer = AutoTokenizer.from_pretrained("Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4")

# Configure LoRA
lora_config = LoraConfig(
    r=16,                        # Rank (grid search: 8, 16, 32)
    lora_alpha=32,               # Scaling factor (grid search: 2*r)
    lora_dropout=0.1,            # Dropout (grid search: 0.05, 0.1, 0.2)
    target_modules=["q_proj", "k_proj", "v_proj", "o_proj", "up_proj", "down_proj", "gate_proj"],
    task_type="CAUSAL_LM",
    bias="none"
)

# Attach LoRA adapters
model = get_peft_model(model, lora_config)
model.print_trainable_parameters()
# Output: trainable params: 8.4M || all params: 3.09B || trainable%: 0.27%
```

### Anti-Patterns to Avoid
- **bf16 on T4**: T4 doesn't support bfloat16. Use `fp16=True` in TrainingArguments, not `bf16=True`.
- **Standard AdamW on T4**: Use `paged_adamw_8bit` optimizer to save VRAM. Standard AdamW exceeds memory with batch_size=4.
- **Ignoring class imbalance**: Without stratification, validation set may lack minority classes (Outdated, Correct), leading to misleading metrics.
- **Transferring suzume adapters**: Different architectures (Llama-3 vs Qwen2.5), incompatible tokenizers. Train from scratch.
- **Manual chat template formatting**: Use `tokenizer.apply_chat_template()` to avoid special token errors.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| LoRA configuration | Manual weight initialization, target module selection | PEFT `LoraConfig` | PEFT handles module discovery, initialization, merging. Custom code prone to errors (wrong modules, init variance). |
| Chat template formatting | String concatenation with `<|im_start|>` tokens | `tokenizer.apply_chat_template()` | Tokenizer knows exact special token IDs, handles edge cases (EOS, padding). Manual formatting causes token ID mismatches. |
| Stratified splitting | Manual class balancing logic | `sklearn.train_test_split(stratify=y)` | scikit-learn handles edge cases (small classes, rounding). Custom logic fails on minority classes. |
| Training loop | Custom gradient accumulation, checkpointing, logging | `SFTTrainer` or `Trainer` | HuggingFace Trainer handles mixed precision, gradient clipping, scheduler, early stopping. Custom loops miss optimizations. |
| Evaluation metrics | Manual F1/accuracy calculation | HuggingFace `evaluate` library | Pre-built metrics handle multi-class edge cases (zero division, macro/micro averaging). |

**Key insight:** QLoRA stack (PEFT + transformers + TRL) is production-hardened by thousands of users. Custom implementations reintroduce solved problems (memory leaks, numerical instability, checkpoint corruption).

## Common Pitfalls

### Pitfall 1: GPTQ + QLoRA Training Instability
**What goes wrong:** Training loss becomes 0, validation loss shows NaN, or dtype errors (`torch.int32` passed to float-only functions).
**Why it happens:** GPTQ models are pre-quantized to int4. Adding QLoRA introduces mixed-precision complexity. Some PEFT versions have dtype bugs with GPTQ.
**How to avoid:**
- Use transformers 4.37.0+ and peft 0.18.1+ (GPTQ support stabilized in these versions).
- Set `torch_dtype="auto"` when loading model (not explicit int32).
- Use `fp16=True` in TrainingArguments, not `bf16=True` (T4 limitation).
- Start with small learning rate (2e-4) to detect instability early.
**Warning signs:**
- Validation loss increases while training loss decreases → overfitting (reduce epochs or add regularization).
- NaN in logs → gradient explosion (reduce learning rate, add gradient clipping).
- OOM errors despite batch_size=4 → disable gradient checkpointing if enabled, or reduce sequence length.

### Pitfall 2: Imbalanced Validation Set Due to Small Dataset
**What goes wrong:** With only 1001 samples and 4 classes, random 80/20 split may leave minority classes (Correct: 50 samples, Outdated: 20 samples) with <5 validation examples. Metrics become unreliable.
**Why it happens:** Small dataset + imbalanced classes + random split = high variance in validation set composition.
**How to avoid:**
- **Always use stratified split**: `train_test_split(stratify=df["Severity Label"])` ensures proportional class distribution.
- **Verify split quality**: Print `val_df["Severity Label"].value_counts()` to confirm minority classes have ≥10 examples.
- **Use class weights (optional)**: If classes are severely imbalanced (Biased: 500 vs Outdated: 20), pass `class_weight="balanced"` to loss function.
**Warning signs:**
- F1 score for minority class is 0 or 1 (too few validation examples to be meaningful).
- Validation accuracy wildly different across epochs (unstable due to small validation set).

### Pitfall 3: Chat Template Formatting Errors
**What goes wrong:** Model produces garbage output or ignores instruction during inference. Training loss plateaus at high value.
**Why it happens:** Qwen2.5 expects exact ChatML format with special tokens (`<|im_start|>`, `<|im_end|>`). Manual formatting misses token IDs or adds extra whitespace.
**How to avoid:**
- **Always use `tokenizer.apply_chat_template()`**: Don't manually concatenate strings.
- **Verify with sample**: Print `tokenizer.decode(tokenizer.encode(formatted_text))` to see actual token sequence.
- **Match training and inference**: Use same template formatting in training script and vLLM inference (Phase 05.2).
**Warning signs:**
- Model repeats prompt instead of generating classification → template mismatch.
- Training loss doesn't decrease below 2.0 after epoch 1 → model not learning instruction format.

### Pitfall 4: Hyperparameter Grid Doesn't Include Optimal Config
**What goes wrong:** All 9 configs achieve <75% validation accuracy. Grid search "fails" because optimal hyperparameters are outside search space.
**Why it happens:** Grid is too narrow (only 3 ranks, 3 dropouts) or wrong scaling rule (alpha = 2*r may not be optimal).
**How to avoid:**
- **Start with research-backed ranges**: Rank [8, 16, 32] covers typical range. Dropout [0.05, 0.1, 0.2] is standard.
- **Use alpha scaling rule**: alpha = 2*r is common heuristic, but if all configs fail, try alpha = r or alpha = 4*r.
- **Monitor trends**: If best config is r=32, consider testing r=64. If r=8 wins, try r=4.
- **Iterate if needed**: If no config reaches 80%, expand grid or adjust learning rate/epochs.
**Warning signs:**
- All configs have similar validation loss (within 0.05) → grid too narrow, not exploring enough.
- Best config is at edge of grid (r=32, d=0.2) → optimal may be outside range.

### Pitfall 5: T4 GPU Memory Management Failures
**What goes wrong:** OOM (Out of Memory) errors despite batch_size=4 and QLoRA. Training crashes partway through.
**Why it happens:** Optimizer states (AdamW) consume 2x model memory. Standard AdamW uses 32-bit states. With r=32 and batch_size=4, memory spikes during backward pass.
**How to avoid:**
- **Use paged_adamw_8bit optimizer**: `optim="paged_adamw_8bit"` in TrainingArguments saves ~40% VRAM.
- **Reduce batch size to 2 if needed**: Use `gradient_accumulation_steps=2` to maintain effective batch size of 4.
- **Monitor GPU usage**: Run `watch -n 1 nvidia-smi` during training to catch memory spikes.
- **Reduce sequence length**: If dataset has very long sentences, truncate to 512 tokens instead of 1024.
**Warning signs:**
- OOM on r=32 but not r=16 → optimizer states are the issue, switch to paged_adamw_8bit.
- Memory usage slowly increases across steps → memory leak (restart training, check for unused tensors).

```python
    eval_metrics = trainer.evaluate()

    # Record results
    results.append({
        "config": config_name,
        "rank": rank,
        "alpha": alpha,
        "dropout": dropout,
        "val_loss": eval_metrics["eval_loss"],
        "val_accuracy": eval_metrics.get("eval_accuracy", 0.0)
    })

    # Save adapter
    trainer.save_model(f"./adapters/{config_name}")

    # Clean up
    del model, trainer
    torch.cuda.empty_cache()

# Find best config
best = min(results, key=lambda x: x["val_loss"])
Path("./adapters/best_config.json").write_text(json.dumps(best, indent=2))

print(f"\n{'='*60}")
print(f"Grid search complete!")
print(f"Best config: {best['config']}")
print(f"  Validation loss: {best['val_loss']:.4f}")
print(f"  Validation accuracy: {best['val_accuracy']:.2%}")
print(f"{'='*60}\n")
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Full fine-tuning entire model | QLoRA (4-bit + LoRA) | 2023 (QLoRA paper) | Enables 3B model training on 16GB GPU. 75% VRAM reduction. |
| Manual hyperparameter tuning | Automated grid search | Ongoing trend | Overnight unattended training tests all configs. Removes human bias. |
| Manual chat formatting | `tokenizer.apply_chat_template()` | transformers 4.34+ (Oct 2023) | Eliminates special token errors. Standardized across models. |
| Standard AdamW | PagedAdamW 8-bit | bitsandbytes 0.41+ (2023) | 40% optimizer VRAM savings. Critical for T4 training. |
| AWQ quantization | GPTQ quantization | For T4 (compute capability 7.5) | AWQ requires compute 8.0+. GPTQ works on older GPUs. |
| Separate train/eval scripts | SFTTrainer (TRL) | TRL 0.4+ (2023) | Unified training with PEFT integration, automatic logging. |

**Deprecated/outdated:**
- **Manual LoRA implementation**: PEFT library is now standard. Custom LoRA code is unmaintained and error-prone.
- **bf16 on T4**: T4 GPU doesn't support bfloat16. Use fp16 instead. bf16 is standard on A100/H100.
- **transformers <4.37.0**: Qwen2.5 architecture added in 4.37.0. Older versions raise `KeyError: 'qwen2'`.
- **Full AdamW on T4**: Use paged_adamw_8bit to avoid OOM errors with batch_size=4.

## Open Questions

1. **Is 1001 samples sufficient for 80% validation accuracy?**
   - What we know: Small datasets risk overfitting. Typical LLM fine-tuning uses 10k+ samples.
   - What's unclear: Whether 3 epochs on 800 samples generalizes well. May need data augmentation.
   - Recommendation: Run pilot as designed. If accuracy <80%, consider data augmentation (Phase 05.6) or semi-supervised learning.

2. **Should we use compute_metrics callback for F1 scores?**
   - What we know: SFTTrainer supports custom metrics via `compute_metrics` function.
   - What's unclear: Whether to implement during grid search or post-hoc analysis.
   - Recommendation: Log F1 scores during training for detailed analysis. Use HuggingFace `evaluate` library for multi-class F1 (macro/micro).

3. **How to handle very long sentences (>512 tokens)?**
   - What we know: Dataset may have academic sentences exceeding 512 tokens. Truncation loses context.
   - What's unclear: Whether to truncate, split, or extend max_seq_length to 1024.
   - Recommendation: Check dataset `df["Sentence"].apply(lambda x: len(tokenizer.encode(x))).describe()`. If 95th percentile >512, use max_seq_length=1024. If >1024, split sentences or truncate.
