**Pilot Phase**: Train LoRA adapter for Qwen2.5-3B-Instruct-GPTQ-Int4 using English-only dataset to validate architecture and find optimal hyperparameters. This phase delivers:
- Grid search across 9 configurations (3 ranks × 3 dropout values)
- Best performing LoRA adapter trained on English data

- Performance baseline for future Hebrew expansion

**Scope Note**: This is an English-only pilot. Hebrew data collection and bilingual training are deferred to a follow-up phase after validating the approach.

<decisions>
## Implementation Decisions

### Training Approach
- Training mode: From scratch (no transfer from suzume adapter)
  - Rationale: Different architectures (Llama-3 8B vs Qwen2.5 3B), different tokenizers, clean start
- Dataset: `data/augmented_dataset.csv` (1001 English sentences)
- Data split: 80/20 train/validation (~800 train, ~200 validation)
- Stratification: Yes — balance severity classes and any other relevant features across splits
- Input format: Qwen2.5 chat template with system/user/assistant tags
  - Leverage instruction-tuned model capabilities
  - Format: system prompt → user (sentence) → assistant (classification JSON)

### Hyperparameter Grid Search
- **Rank (r)**: [8, 16, 32]
- **Alpha**: Scale with rank using alpha = 2*r
  - r=8 → alpha=16
  - r=16 → alpha=32
  - r=32 → alpha=64
- **Dropout**: [0.05, 0.1, 0.2]
- **Total configurations**: 9 training runs (3 ranks × 3 dropout values)
- **Execution**: Sequential with automation — script loops through all configs overnight
- **Output**: Each config saved to separate directory (e.g., `qwen_r8_d0.05/`, `qwen_r16_d0.1/`, etc.)

### Training Environment
- **Location**: Azure VM (InclusifyModel) with T4 GPU
- **Method**: QLoRA (4-bit quantized training) with GPTQ base model
- **Epochs**: 3 per config
- **Batch size**: 4 (safe for T4 16GB VRAM across all rank values)
- **Precision**: fp16 for adapters (T4 doesn't support bf16)
- **Target modules**: Keep suzume config — [q_proj, k_proj, v_proj, o_proj, up_proj, down_proj, gate_proj]
- **Estimated time**: ~1 hour per config × 9 configs = 9-10 hours total

### Evaluation Strategy
- **Metrics tracked**:
  - Validation loss (primary selection criterion)
  - Validation accuracy
  - Per-class F1 scores (Outdated, Biased, Offensive, Incorrect)
- **Config selection**: Best validation loss after 3 epochs
- **Acceptance criteria**: Validation accuracy > 80%
  - Pilot success threshold — proves Qwen2.5 + QLoRA works before scaling to Hebrew
- **Logging**: Save all metrics for all 9 configs for comparison and analysis

### Claude's Discretion
- Exact Python libraries for QLoRA training (peft, transformers versions compatible with Qwen2.5)
- Training script structure and logging format
- Exact chat template formatting details from Qwen2.5 tokenizer
- Learning rate and optimizer choice (AdamW with standard schedule typical)
- Gradient accumulation if needed for memory management
- Warmup steps and scheduler type

</decisions>
