# Phase 05.3: Auth Frontend - Research

**Researched:** 2026-03-11
**Domain:** Frontend authentication (Next.js 16, React 19, OAuth, JWT)
**Confidence:** HIGH

## Summary

This phase implements frontend authentication UI for the Inclusify platform. The backend already has FastAPI Users configured with JWT auth (`/auth/jwt/login`, `/auth/jwt/register`) and the infrastructure supports Google OAuth via httpx-oauth. The frontend needs login/register pages, an AuthContext for state management, and Google OAuth integration.

**Key insight:** Since FastAPI Users already handles JWT tokens via BearerTransport, the frontend should manage tokens directly without NextAuth.js overhead. A custom AuthContext with fetch-based API calls is simpler and matches the existing backend contract.

**Primary recommendation:** Build custom AuthContext with localStorage token storage (Claude's discretion per CONTEXT.md), add Google OAuth via backend's OAuth router, use react-hook-form + Zod for validation, and Sonner for toast notifications.

### Pattern 1: AuthContext with Token Management
**What:** React Context that manages JWT tokens, user state, and auth operations
**When to use:** All authenticated operations across the app
**Example:**
```typescript
// contexts/AuthContext.tsx
'use client';

import { createContext, useContext, useState, useEffect, ReactNode } from 'react';

interface User {
  id: string;
  email: string;
  role: string;
}

interface AuthContextType {
  user: User | null;
  isLoading: boolean;
  login: (email: string, password: string, rememberMe: boolean) => Promise<void>;
  register: (email: string, password: string) => Promise<void>;
  logout: () => void;
  getToken: () => string | null;
}

const AuthContext = createContext<AuthContextType | null>(null);

export function AuthProvider({ children }: { children: ReactNode }) {
  const [user, setUser] = useState<User | null>(null);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    // Check for existing token on mount
    const token = localStorage.getItem('auth_token');
    if (token) {
      validateAndSetUser(token);
    } else {
      setIsLoading(false);
    }
  }, []);

  // ... implementation
}

export function useAuth() {
  const context = useContext(AuthContext);
  if (!context) throw new Error('useAuth must be used within AuthProvider');
  return context;
}
```

### Pattern 2: Form with react-hook-form + Zod
**What:** Type-safe form validation with inline errors
**When to use:** Login and register forms
**Example:**
```typescript
// components/auth/LoginForm.tsx
'use client';

import { useForm } from 'react-hook-form';
import { zodResolver } from '@hookform/resolvers/zod';
import { z } from 'zod';

const loginSchema = z.object({
  email: z.string().email('Invalid email address'),
  password: z.string().min(1, 'Password is required'),
  rememberMe: z.boolean().default(false),
});

type LoginFormData = z.infer<typeof loginSchema>;

export function LoginForm() {
  const { register, handleSubmit, formState: { errors } } = useForm<LoginFormData>({
    resolver: zodResolver(loginSchema),
  });

  return (
    <form onSubmit={handleSubmit(onSubmit)}>
      <input {...register('email')} />
      {errors.email && <span className="text-red-500 text-sm">{errors.email.message}</span>}
      {/* ... */}
    </form>
  );
}
```

### Pattern 4: Protected Routes with AuthGuard
**What:** Component that checks auth state and redirects unauthorized users
**When to use:** Admin page protection
**Example:**
```typescript
// components/auth/AuthGuard.tsx
'use client';

import { useAuth } from '@/contexts/AuthContext';
import { notFound } from 'next/navigation';

export function AdminGuard({ children }: { children: React.ReactNode }) {
  const { user, isLoading } = useAuth();

  if (isLoading) return <LoadingSpinner />;
  if (!user || user.role !== 'site_admin') {
    notFound(); // Returns 404 as per CONTEXT.md
  }

  return <>{children}</>;
}
```

### Anti-Patterns to Avoid
- **Storing tokens in cookies without httpOnly:** Exposes tokens to XSS. If using cookies, backend must set httpOnly.
- **JWT verification on frontend:** Never validate JWT signature client-side; trust backend's /users/me endpoint.
- **Calling logout without clearing all state:** Must clear localStorage AND context state.
- **Hardcoding redirect URLs:** Use environment variables for OAuth callback URLs.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Form validation | Custom validation logic | react-hook-form + Zod | Edge cases (async validation, touched states, error mapping) |
| Toast notifications | Custom toast system | Sonner | Animation, stacking, dismiss logic, accessibility |
| Dropdown menus | Custom dropdown | @radix-ui/react-dropdown-menu | Focus management, keyboard nav, positioning |
| Google button styling | Custom SVG button | Google's branding guidelines | Must follow official guidelines for app verification |
| Token decode | Custom JWT parser | Simple atob + JSON.parse | JWT payload is just base64, no crypto needed on frontend |

**Key insight:** Form libraries handle dozens of edge cases (blur vs submit validation, dirty tracking, async validation, error display timing) that take weeks to implement correctly.

## Common Pitfalls

### Pitfall 1: Token Expiration Handling
**What goes wrong:** User gets 401, app crashes or shows generic error
**Why it happens:** No global error interceptor for expired tokens
**How to avoid:** Create fetch wrapper that catches 401, shows toast, clears auth state, redirects to login with return URL
**Warning signs:** Users reporting "random logouts" or "white screens"

### Pitfall 2: OAuth Callback State Loss
**What goes wrong:** After Google OAuth, user loses their intended destination
**Why it happens:** State param not preserved through OAuth flow
**How to avoid:** Store intended URL in localStorage before OAuth redirect, restore after callback
**Warning signs:** Users always landing on home page after Google login

### Pitfall 3: RTL Layout Breaking
**What goes wrong:** Form layout breaks in Hebrew mode
**Why it happens:** Not using logical properties (start/end vs left/right)
**How to avoid:** Use Tailwind's RTL utilities (ms-, me-, ps-, pe-) and test both directions
**Warning signs:** Buttons and labels misaligned in Hebrew

### Pitfall 4: Flash of Unauthenticated Content
**What goes wrong:** Protected content briefly shows before redirect
**Why it happens:** Auth check happens after initial render
**How to avoid:** Show loading state while auth is being verified; check isLoading before rendering content
**Warning signs:** Users see admin panel briefly before being redirected

### Pitfall 5: Inconsistent Error Messages
**What goes wrong:** Backend sends different error formats than frontend expects
**Why it happens:** FastAPI Users has specific error response shapes
**How to avoid:** Map backend errors to user-friendly messages; handle both detail string and detail array formats
**Warning signs:** Users seeing "undefined" or JSON in error messages

## Code Examples

### FastAPI Users Login Request
```typescript
// lib/api/auth.ts
const API_BASE_URL = process.env.NEXT_PUBLIC_API_URL || 'http://localhost:8000';

export async function login(email: string, password: string): Promise<{ access_token: string }> {
  // FastAPI Users expects form-urlencoded for login
  const response = await fetch(`${API_BASE_URL}/auth/jwt/login`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/x-www-form-urlencoded',
    },
    body: new URLSearchParams({
      username: email, // FastAPI Users uses 'username' for email
      password,
    }),
  });

  if (!response.ok) {
    const error = await response.json();
    throw new Error(error.detail || 'Login failed');
  }

  return response.json();
}
```

### FastAPI Users Register Request
```typescript
// lib/api/auth.ts
export async function register(email: string, password: string): Promise<User> {
  const response = await fetch(`${API_BASE_URL}/auth/jwt/register`, {
    method: 'POST',
    headers: {
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({ email, password }),
  });

  if (!response.ok) {
    const error = await response.json();
    // Handle FastAPI Users validation errors
    if (error.detail === 'REGISTER_USER_ALREADY_EXISTS') {
      throw new Error('An account with this email already exists');
    }
    throw new Error(error.detail || 'Registration failed');
  }

  return response.json();
}
```

### Avatar with Initials Fallback
```typescript
// components/auth/UserAvatar.tsx
function hashCode(str: string): number {
  let hash = 0;
  for (let i = 0; i < str.length; i++) {
    hash = str.charCodeAt(i) + ((hash << 5) - hash);
  }
  return hash;
}

const AVATAR_COLORS = [
  'bg-red-500', 'bg-orange-500', 'bg-amber-500', 'bg-yellow-500',
  'bg-lime-500', 'bg-green-500', 'bg-emerald-500', 'bg-teal-500',
  'bg-cyan-500', 'bg-sky-500', 'bg-blue-500', 'bg-indigo-500',
  'bg-violet-500', 'bg-purple-500', 'bg-fuchsia-500', 'bg-pink-500',
];

export function UserAvatar({ email }: { email: string }) {
  const initials = email.slice(0, 2).toUpperCase();
  const colorIndex = Math.abs(hashCode(email)) % AVATAR_COLORS.length;
  const bgColor = AVATAR_COLORS[colorIndex];

  return (
    <div className={`w-8 h-8 rounded-full ${bgColor} flex items-center justify-center text-white text-sm font-medium`}>
      {initials}
    </div>
  );
}
```

### Toast on 401 (Session Expired)
```typescript
// lib/api/client.ts (update existing file)
import { toast } from 'sonner';

export async function fetchWithAuth(url: string, options: RequestInit = {}) {
  const token = localStorage.getItem('auth_token');

  const response = await fetch(url, {
    ...options,
    headers: {
      ...options.headers,
      ...(token ? { Authorization: `Bearer ${token}` } : {}),
    },
  });

  if (response.status === 401) {
    // Clear auth state
    localStorage.removeItem('auth_token');

    // Show toast
    toast.error('Session expired. Please log in again.');

    // Redirect with return URL
    const returnUrl = encodeURIComponent(window.location.pathname);
    window.location.href = `/login?returnUrl=${returnUrl}`;

    throw new Error('Session expired');
  }

  return response;
}
```

- `passlib` in FastAPI Users: Replaced by `pwdlib` (already done in backend)

## Open Questions

1. **Google OAuth Backend Setup**
   - What we know: FastAPI Users supports httpx-oauth, backend has auth routers
   - What's unclear: Is Google OAuth router already configured in backend?
   - Recommendation: Check if `/auth/google/authorize` endpoint exists; if not, backend may need OAuth router added first

2. **Token Refresh Strategy**
   - What we know: Backend has Redis configured for refresh tokens
   - What's unclear: Whether refresh token endpoint is exposed and format
   - Recommendation: Start with short-lived access tokens only; add refresh later if needed

3. **Remember Me Implementation**
   - What we know: User wants 30 days vs session-only
   - What's unclear: Backend token lifetime configuration
   - Recommendation: Frontend can request different token lifetimes; backend must support configurable expiry
