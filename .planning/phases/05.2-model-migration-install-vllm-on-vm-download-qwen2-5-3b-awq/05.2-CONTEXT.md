<decisions>
## Implementation Decisions

### vLLM Installation
- Installation method: pip in Python 3.11 venv (isolated, easy to manage)
- Python version: 3.11 (install via pyenv or deadsnakes if not present)
- Service type: Systemd service — auto-start on boot, auto-restart on crash
- Matches Phase 3 decision for production deployment pattern

### Model Download
- Download method: HuggingFace Hub direct (huggingface-cli download or vLLM auto-download)
- Model ID: `Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4` (~2GB)
  - **Updated per research:** Original decision was AWQ, but AWQ is incompatible with T4 GPU (compute capability 7.5, requires 8.0+). GPTQ-Int4 provides equivalent quantization benefits and is fully T4-compatible.
- Storage location: `/home/azureuser/models/` (user home, persists across reboots)

### vLLM Configuration
- Max context length: 2048 tokens (sufficient for sentence-by-sentence analysis, saves VRAM)
- dtype: half (T4 does not support bfloat16, must use float16)
  - **Updated per research:** Original decision was "auto", but T4 requires explicit `--dtype half`.
- Port: 8001 (matches Phase 3 decision, backend expects this)
- Network: localhost only (internal network via VNet, no public exposure)

### LoRA & Verification
- LoRA handling: Test base model only — Phase 5.4 handles LoRA retraining for Qwen2.5
- Verification approach: curl health check + sample inference via OpenAI-compatible API
- Test prompts: Three tests — simple "Hello", Hebrew sentence, English LGBTQ+ sentence

### Claude's Discretion
- Exact vLLM version (0.6.x compatible with CUDA 12.4 and GPTQ)
- pyenv/deadsnakes choice for Python 3.11 installation
- Systemd service file details (ExecStart, restart policy, logging)
- Specific Hebrew and English test sentences
- CUDA toolkit verification steps

</decisions>
