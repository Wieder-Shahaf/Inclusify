# Phase 05.2: Model Migration - Research

**Researched:** 2026-03-11
**Domain:** vLLM deployment, Qwen2.5 model, T4 GPU quantization
**Confidence:** HIGH

## Summary

Phase 05.2 installs vLLM on the InclusifyModel Azure VM and downloads the Qwen2.5-3B-Instruct model for inference. The T4 GPU on this VM has compute capability 7.5, which creates a critical constraint: **AWQ quantization is NOT supported on T4** (requires compute capability 8.0+). The CONTEXT.md specifies Qwen2.5-3B-Instruct-AWQ, but this will fail at runtime.

The recommended solution is to use **Qwen2.5-3B-Instruct-GPTQ-Int4** instead, which is fully supported on T4 GPUs and provides similar memory savings (~2GB model size vs ~6GB fp16). GPTQ has been the established pattern for T4 deployment throughout the project (Phase 3 research also recommended GPTQ for T4). vLLM 0.6.x+ fully supports GPTQ quantization with automatic kernel selection.

**Primary recommendation:** Use `Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4` instead of AWQ variant. Install vLLM 0.6.6 (stable, compatible with CUDA 12.4 on the VM). Deploy as systemd service on port 8001 with `--dtype=half` (T4 does not support bfloat16).

## CRITICAL: AWQ Incompatibility with T4 GPU

**This is the most important finding of this research.**

The CONTEXT.md specifies downloading `Qwen/Qwen2.5-3B-Instruct-AWQ`, but **AWQ quantization does NOT work on T4 GPUs**:

| GPU | Compute Capability | AWQ Support |
|-----|-------------------|-------------|
| T4 (InclusifyModel VM) | 7.5 | NOT SUPPORTED |
| RTX 30xx | 8.6 | Supported |
| A10/A100 | 8.0+ | Supported |

**Error that will occur if AWQ is used:**
```
RuntimeError: Quantization scheme is not supported for the current GPU.
Min capability: 80. Current capability: 75.
```

**Resolution:** Use `Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4` instead. This provides:
- Same 4-bit quantization (~2GB model size)
- Full T4 GPU support
- vLLM automatic GPTQ kernel selection
- Official Qwen model (not third-party quantization)

Sources:
- [vLLM Issue #25022](https://github.com/vllm-project/vllm/issues/25022) - AWQ compute capability requirement
- [HuggingFace Qwen2.5-3B-Instruct-GPTQ-Int4](https://huggingface.co/Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4) - Official model

## Architecture Patterns

### Recommended Directory Structure (VM)

### Pattern 1: Model Download via huggingface-cli
**What:** Download model to local directory for persistent storage
**When to use:** One-time setup, ensures model persists across reboots
**Example:**
```bash
# Set custom cache directory
export HF_HOME=/home/azureuser/models

# Download model
huggingface-cli download Qwen/Qwen2.5-3B-Instruct-GPTQ-Int4 \
    --local-dir /home/azureuser/models/Qwen2.5-3B-Instruct-GPTQ-Int4
```

### Pattern 2: vLLM Serve Command for GPTQ
**What:** Launch vLLM with correct parameters for T4 GPU
**When to use:** Starting vLLM server manually or via systemd
**Example:**
```bash
# Source: vLLM docs + Phase 3 research
vllm serve /home/azureuser/models/Qwen2.5-3B-Instruct-GPTQ-Int4 \
    --dtype half \
    --max-model-len 2048 \
    --port 8001 \
    --host 127.0.0.1 \
    --gpu-memory-utilization 0.9
```

### Pattern 3: Systemd Service for vLLM
**What:** Auto-start vLLM on boot, auto-restart on crash
**When to use:** Production deployment
**Example:**
```ini
# /etc/systemd/system/vllm.service
[Unit]
Description=vLLM Inference Server (Qwen2.5-3B-GPTQ)
After=network.target

[Service]
Type=simple
User=azureuser
WorkingDirectory=/home/azureuser
Environment="CUDA_VISIBLE_DEVICES=0"
Environment="PATH=/home/azureuser/inclusify/.venv/bin:/usr/local/cuda-12.4/bin:/usr/bin"
Environment="LD_LIBRARY_PATH=/usr/local/cuda-12.4/lib64"
ExecStart=/home/azureuser/inclusify/.venv/bin/vllm serve \
    /home/azureuser/models/Qwen2.5-3B-Instruct-GPTQ-Int4 \
    --dtype half \
    --max-model-len 2048 \
    --port 8001 \
    --host 127.0.0.1 \
    --gpu-memory-utilization 0.9
Restart=always
RestartSec=10
StandardOutput=journal
StandardError=journal

[Install]
WantedBy=multi-user.target
```

**When to use:** After installation and service start

### Anti-Patterns to Avoid
- **Using AWQ model on T4:** Will fail with compute capability error. Use GPTQ-Int4.
- **Using --dtype=bfloat16 on T4:** T4 does not support bfloat16. Use `--dtype=half`.
- **Binding to 0.0.0.0:** Exposes vLLM to public network. Use `--host 127.0.0.1`.
- **Using latest vLLM (0.17.0):** Requires CUDA 12.9+. VM has CUDA 12.4, use 0.6.6.
- **Skipping model download:** vLLM auto-download works but is slower on first request and may timeout.

## Common Pitfalls

### Pitfall 1: AWQ on T4 GPU
**What goes wrong:** vLLM crashes with "Min capability: 80. Current capability: 75"
**Why it happens:** AWQ kernels require Ampere+ GPUs (compute capability 8.0+). T4 is Turing (7.5).
**How to avoid:** Use GPTQ-Int4 model instead of AWQ.
**Warning signs:** Immediate crash on vLLM startup when loading AWQ model.

### Pitfall 2: Wrong vLLM Version
**What goes wrong:** pip install vllm installs 0.17.0, which requires CUDA 12.9
**Why it happens:** Default pip install gets latest version
**How to avoid:** Pin version: `pip install vllm==0.6.6`
**Warning signs:** "CUDA version mismatch" or "CUDA capability not found" errors.

### Pitfall 3: bfloat16 on T4
**What goes wrong:** vLLM crashes or produces garbage output
**Why it happens:** T4 (Turing) does not support bfloat16, requires Ampere+ (compute capability 8.0+)
**How to avoid:** Use `--dtype=half` (float16) explicitly.
**Warning signs:** NaN outputs, CUDA errors about unsupported dtype.

### Pitfall 4: Model Not Pre-Downloaded
**What goes wrong:** First request to vLLM takes 10+ minutes and may timeout
**Why it happens:** vLLM auto-downloads model on first request if not cached
**How to avoid:** Pre-download model with huggingface-cli before starting service.
**Warning signs:** Systemd service hangs on start, first API request times out.

### Pitfall 5: Incorrect Model Path in systemd
**What goes wrong:** vLLM fails to find model, exits immediately
**Why it happens:** Systemd runs as azureuser but may not expand ~ or have correct PATH
**How to avoid:** Use absolute paths: `/home/azureuser/models/...`
**Warning signs:** Service fails to start, journalctl shows "model not found".

### Pitfall 6: vLLM Listening on Wrong Interface
**What goes wrong:** Backend can connect but gets connection refused
**Why it happens:** vLLM bound to 127.0.0.1 but backend tries to connect via private IP
**How to avoid:** For VNet access, may need to bind to 0.0.0.0 or specific private IP (but then add firewall rules)
**Warning signs:** Backend logs show "connection refused" to VM private IP.
