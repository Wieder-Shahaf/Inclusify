# Phase 05.5: Backend OAuth - Add Google OAuth endpoints to FastAPI - Research

**Researched:** 2026-03-11
**Domain:** OAuth 2.0 / Google Identity / FastAPI Users OAuth extension
**Confidence:** HIGH

## Summary

Phase 05.5 extends the existing FastAPI Users authentication with Google OAuth support. The project already uses FastAPI Users 13.0.0 with a custom JWT strategy (JWTStrategyWithRole) that includes role claims. The frontend has a "Sign in with Google" button (Phase 05.3) that redirects to `/auth/google/authorize` and expects token back via `/oauth/callback?access_token=xxx`.

FastAPI Users provides built-in OAuth support via the `httpx-oauth` library with `get_oauth_router()`. This handles the full OAuth flow including CSRF state management, code exchange, and user creation/linking. The main implementation work involves: (1) adding OAuthAccount model, (2) configuring GoogleOAuth2 client, (3) mounting OAuth router, (4) customizing callback redirect to match frontend expectations.

**Primary recommendation:** Use FastAPI Users' built-in OAuth router with `httpx-oauth`, configure `associate_by_email=True` and `is_verified_by_default=True` per user decisions, and customize the callback response to redirect to frontend `/oauth/callback?access_token=xxx`.

## User Constraints (from CONTEXT.md)

### Locked Decisions
- Backend-only flow: Frontend redirects to `/auth/google/authorize` -> Google -> Backend callback
- Authorization code exchange happens entirely on backend (more secure)
- After success, redirect to frontend with JWT in URL (e.g., `?access_token=xxx`)
- Frontend extracts token from URL param and stores in localStorage
- Callback URL: `/auth/google/callback` (clean RESTful path, standard FastAPI Users pattern)
- Auto-link accounts: If Google email matches existing password account, link automatically
- User can then use either Google or password to log in
- Trust Google's email verification: Mark user as verified if Google says email is verified
- Identical JWT format to password login (user_id, role, exp claims)
- Same token expiry as password login (use ACCESS_TOKEN_EXPIRE_MINUTES setting)
- Don't store Google's access/refresh tokens - only use Google for identity
- Redirect to frontend with error query param on failure (e.g., `?error=access_denied`)
- Specific error codes: `access_denied`, `invalid_code`, `email_mismatch`, `account_disabled`
- Log OAuth failures with details (error type, timestamp, sanitized context)
- Apply same rate limiting as other auth endpoints

- Exact redirect URL handling for different environments (dev/prod)

### Pattern 1: FastAPI Users OAuth Router
**What:** Use `get_oauth_router()` to generate authorize/callback endpoints
**When to use:** Standard OAuth flow with FastAPI Users
**Example:**
```python
# backend/app/auth/oauth.py
import os
from httpx_oauth.clients.google import GoogleOAuth2

google_oauth_client = GoogleOAuth2(
    client_id=os.getenv("GOOGLE_CLIENT_ID", ""),
    client_secret=os.getenv("GOOGLE_CLIENT_SECRET", ""),
)

# backend/app/auth/users.py (add to existing)
from app.auth.oauth import google_oauth_client

google_oauth_router = fastapi_users.get_oauth_router(
    google_oauth_client,
    auth_backend,
    state_secret=settings.JWT_SECRET,  # Use existing secret for CSRF state
    associate_by_email=True,           # Auto-link by email (user decision)
    is_verified_by_default=True,       # Trust Google's email verification
)

# backend/app/main.py
app.include_router(
    google_oauth_router,
    prefix="/auth/google",
    tags=["Auth"],
)
```

### Pattern 3: Custom OAuth Callback (Frontend Redirect)
**What:** Override default callback behavior to redirect to frontend with token
**When to use:** SPA frontend needs token in URL
**Example:**
```python
# backend/app/auth/oauth_callback.py

from fastapi.responses import RedirectResponse
from httpx_oauth.integrations.fastapi import OAuth2AuthorizeCallback

    request: Request,

    user_manager = Depends(get_user_manager),
```

### Anti-Patterns to Avoid
- **Storing Google tokens:** Decision explicitly says don't store - only use for identity
- **Frontend code exchange:** Doing code exchange in frontend exposes client_secret
- **Missing CSRF state:** Always validate state parameter to prevent CSRF attacks
- **Hardcoded redirect URIs:** Use environment variables for dev/prod flexibility

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| OAuth state management | Custom CSRF state generation/validation | FastAPI Users built-in (`state_secret` param) | Handles secure state cookie, validation, timing |
| Google OAuth endpoints | Manual authorization URL construction | `GoogleOAuth2` from httpx-oauth | Pre-configured Google endpoints, scope handling |
| Token exchange | Raw HTTP POST to Google token endpoint | `get_access_token()` from httpx-oauth | Error handling, response parsing |
| User creation/linking | Custom email lookup + create logic | FastAPI Users `associate_by_email=True` | Handles edge cases, atomic operations |

**Key insight:** OAuth has many security pitfalls (CSRF, token leakage, race conditions in account linking). FastAPI Users + httpx-oauth handle these correctly.

## Common Pitfalls

### Pitfall 1: CSRF State Cookie Not Working in Development
**What goes wrong:** State validation fails because cookie not set/read properly
**Why it happens:** Secure cookies require HTTPS, but localhost uses HTTP
**How to avoid:** Set `csrf_token_cookie_secure=False` for local development
**Warning signs:** "Invalid state" errors during OAuth callback

### Pitfall 2: Redirect URI Mismatch
**What goes wrong:** Google returns `redirect_uri_mismatch` error
**Why it happens:** Callback URL doesn't exactly match Google Console config
**How to avoid:** Register both `http://localhost:8000/auth/google/callback` AND production URL in Google Console
**Warning signs:** OAuth flow fails immediately after Google consent

### Pitfall 3: Missing OAuthAccount Relationship
**What goes wrong:** OAuth login creates user but can't find them on subsequent logins
**Why it happens:** User model missing `oauth_accounts` relationship with `lazy="joined"`
**How to avoid:** Always include the relationship with eager loading
**Warning signs:** "User already exists" errors on every OAuth login

### Pitfall 4: Frontend Callback URL Mismatch
**What goes wrong:** Frontend expects `?access_token=xxx` but backend sends different format
**Why it happens:** Default FastAPI Users callback returns JSON, not redirect
**How to avoid:** Custom callback endpoint that redirects with token in query param
**Warning signs:** Frontend callback page shows "Completing sign in..." forever

### Pitfall 5: JWT Without Role Claim
**What goes wrong:** OAuth users get JWT without role, frontend shows wrong permissions
**Why it happens:** Default OAuth callback doesn't use custom JWTStrategyWithRole
**How to avoid:** Ensure OAuth callback uses same `JWTStrategyWithRole.write_token()` as password login
**Warning signs:** OAuth users see different UI than password users

### OAuth Router with Custom Redirect
```python
# Custom implementation for frontend redirect requirement
from fastapi import APIRouter, Depends, Request, HTTPException
from fastapi.responses import RedirectResponse
from starlette.status import HTTP_302_FOUND
import logging

logger = logging.getLogger(__name__)

@router.get("/callback")
async def oauth_callback(
    request: Request,
    code: str = None,
    state: str = None,
    error: str = None,
):
    frontend_url = settings.FRONTEND_URL or "http://localhost:3000"
    callback_path = "/en/oauth/callback"  # Match frontend route

    # Handle OAuth errors
    if error:
        logger.warning(f"OAuth error: {error}")
        return RedirectResponse(
            url=f"{frontend_url}{callback_path}?error={error}",
            status_code=HTTP_302_FOUND,
        )

    # ... validate state, exchange code, create/get user ...

    # Generate JWT using existing strategy
    jwt_strategy = get_jwt_strategy()
    token = await jwt_strategy.write_token(user)

    return RedirectResponse(
        url=f"{frontend_url}{callback_path}?access_token={token}",
        status_code=HTTP_302_FOUND,
    )
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| passlib for password hashing | pwdlib[argon2] | FastAPI Users 13.0 | Already using pwdlib, no change needed |
| Implicit OAuth flow | Authorization code flow | OAuth 2.1 spec | More secure, backend handles code exchange |
| Token in URL hash (#) | Token in query param (?) | Current best practice | Query params work with SSR, hash doesn't |

**Deprecated/outdated:**
- Implicit flow (response_type=token): Deprecated in OAuth 2.1, use authorization code
- Storing OAuth provider's access tokens: Only needed if calling provider APIs post-auth

## Open Questions

1. **Locale in redirect URL**
   - What we know: Frontend callback is at `/{locale}/oauth/callback`
   - What's unclear: How to determine user's locale for redirect (en vs he)
   - Recommendation: Default to `en`, or store locale in state parameter

2. **Database migration for oauth_accounts table**
   - What we know: Need new table for OAuthAccount model
   - What's unclear: Whether to use SQLAlchemy auto-create or explicit migration
   - Recommendation: Let `create_db_and_tables()` handle it (already called in lifespan)
