# Phase 05.5: Backend OAuth - Context

**Gathered:** 2026-03-11
**Status:** Ready for planning

<domain>
## Phase Boundary

Add Google OAuth endpoints to the FastAPI backend. The frontend (Phase 05.3) has a "Sign in with Google" button that needs backend routes to complete the OAuth flow. FastAPI Users is already configured for email/password auth — this phase extends it with Google OAuth support. Deliverables:
- `/auth/google/authorize` endpoint to initiate OAuth flow
- `/auth/google/callback` endpoint to handle Google's redirect
- Account creation/linking logic for OAuth users
- JWT generation after successful OAuth

</domain>

<decisions>
## Implementation Decisions

### OAuth Flow Architecture
- Backend-only flow: Frontend redirects to `/auth/google/authorize` → Google → Backend callback
- Authorization code exchange happens entirely on backend (more secure)
- After success, redirect to frontend with JWT in URL hash fragment (e.g., `/#token=xxx`)
- Frontend extracts token from hash and stores in localStorage
- Callback URL: `/auth/google/callback` (clean RESTful path, standard FastAPI Users pattern)

### Account Linking Behavior
- Auto-link accounts: If Google email matches existing password account, link automatically
- User can then use either Google or password to log in
- Trust Google's email verification: Mark user as verified if Google says email is verified
- Account linking management (unlink Google): Defer to later phase

### Token Response Format
- Identical JWT format to password login (user_id, role, exp claims)
- Same token expiry as password login (use ACCESS_TOKEN_EXPIRE_MINUTES setting)
- "Remember me" handled by frontend token storage, not backend
- Don't store Google's access/refresh tokens — only use Google for identity, not API access

### Error Handling Strategy
- Redirect to frontend with error query param on failure (e.g., `?error=access_denied`)
- Specific error codes: `access_denied`, `invalid_code`, `email_mismatch`, `account_disabled`
- Frontend shows appropriate error message based on code
- Log OAuth failures with details (error type, timestamp, sanitized context)
- Apply same rate limiting as other auth endpoints

### Claude's Discretion
- CSRF state parameter implementation details
- Google OAuth library choice (httpx-oauth, authlib, or raw httpx)
- Exact redirect URL handling for different environments (dev/prod)
- Token hash fragment encoding

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `backend/app/auth/backend.py`: JWTStrategyWithRole already generates JWTs with role claim — reuse for OAuth
- `backend/app/auth/users.py`: FastAPIUsers instance and auth_backend available for extension
- `backend/app/auth/manager.py`: UserManager handles user lifecycle events — add on_after_oauth_login
- `backend/app/db/models.py`: User model may need oauth_accounts relationship
- `backend/app/core/config.py`: Add GOOGLE_CLIENT_ID and GOOGLE_CLIENT_SECRET settings

### Established Patterns
- Auth routers mounted at `/auth/jwt/*` — OAuth can follow `/auth/google/*`
- JWT generation via JWTStrategyWithRole.write_token()
- User creation via FastAPI Users' create_user flow
- Logging via Python logging module (logger.info for auth events)

### Integration Points
- `backend/app/main.py`: Mount new google_oauth_router
- `backend/app/auth/__init__.py`: Export OAuth router
- Google Cloud Console: Configure OAuth credentials with callback URL
- Environment variables: GOOGLE_CLIENT_ID, GOOGLE_CLIENT_SECRET

</code_context>

<specifics>
## Specific Ideas

- Use same JWT claims structure (sub, role, aud, exp) for OAuth logins
- Frontend Phase 05.3 has Google button ready — just needs backend to respond
- Callback should work with both localhost and production domains
- Follow Google's OAuth 2.0 for Web Server Applications flow

</specifics>
