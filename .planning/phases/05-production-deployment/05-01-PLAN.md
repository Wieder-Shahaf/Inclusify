  <name>Task 1: Extend azure-setup.sh with Container Apps Environment</name>
  <files>infra/scripts/azure-setup.sh</files>
  <action>
Extend the existing azure-setup.sh script to create a Container Apps Environment with VNet integration.

Add after PostgreSQL creation (step 4):

1. Create Container Apps subnet in vLLM VNet:
   - VNet: InclusifyModel-vnet in Group07 (existing vLLM network)
   - Subnet name: containerapps-subnet
   - Address prefix: 10.0.2.0/27 (minimum /27 required for workload profiles)
   - Delegate to Microsoft.App/environments

2. Create Container Apps Environment:
   - Name: inclusify-env
   - Resource group: inclusify-rg (same as app resources)
   - Use infrastructure-subnet-resource-id from step 1

3. Get and store Container Apps outbound IPs:
   - Query: az containerapp env show --query "properties.outboundIpAddresses"
   - Display for PostgreSQL firewall configuration

4. Add Container Apps outbound IPs to PostgreSQL firewall:
   - Loop through outbound IPs
   - Create firewall rule for each: az postgres flexible-server firewall-rule create
   - IMPORTANT: PostgreSQL is in inclusify-rg (not Group07), so use:
     `az postgres flexible-server firewall-rule create --resource-group inclusify-rg --server-name inclusify-db ...`

Script must remain idempotent (use || echo patterns for existing resources).

Note: The vLLM VNet is in Group07 but Container Apps Environment and PostgreSQL are in inclusify-rg.
This works because we reference the subnet by resource ID across resource groups.
  </action>

  </verify>

Create a new script to deploy/update Container Apps. Script should:

1. Accept IMAGE_TAG parameter (default: latest, use git SHA in CI)

2. Build and push images:
   - Build backend: docker build -f infra/docker/backend.Dockerfile --target runtime
   - Build frontend: docker build -f infra/docker/frontend.Dockerfile --target runner
   - Tag with ACR name and IMAGE_TAG
   - Push to ACR (reuse azure-push.sh pattern)

3. Deploy/update backend Container App:
   - Name: inclusify-backend
   - Image: inclusifyacr.azurecr.io/inclusify-backend:${IMAGE_TAG}
   - Target port: 8000
   - Ingress: external (public URL)
   - minReplicas: 1 (prevent cold starts per PITFALLS.md #11)
   - maxReplicas: 3
   - Secrets (use az containerapp secret set):
     - db-password: ${PG_PASSWORD}
     - jwt-secret: ${JWT_SECRET}
   - Environment variables:
     - PGHOST=inclusify-db.postgres.database.azure.com
     - PGPORT=5432
     - PGDATABASE=inclusify
     - PGUSER=inclusifyadmin
     - PGPASSWORD=secretref:db-password
     - PGSSL=require
     - JWT_SECRET=secretref:jwt-secret
     - VLLM_URL=http://10.0.1.4:8001 (vLLM VM private IP, confirm from VM setup)
     - ALLOWED_ORIGINS=https://${FRONTEND_FQDN} (set after frontend deploys)
   - Health probes:
     - Startup probe: /health, initial delay 10s, period 5s, failure threshold 30
     - Liveness probe: /health, period 30s

4. Deploy/update frontend Container App:
   - Name: inclusify-frontend
   - Image: inclusifyacr.azurecr.io/inclusify-frontend:${IMAGE_TAG}
   - Target port: 3000
   - Ingress: external

   - Environment variables:
     - NEXT_PUBLIC_API_URL=https://${BACKEND_FQDN}
     - NEXT_PUBLIC_USE_DEMO_MODE=false

5. Update backend ALLOWED_ORIGINS after frontend FQDN is known:
   - Get frontend FQDN: az containerapp show --query properties.configuration.ingress.fqdn
   - Update backend env var: az containerapp update --set-env-vars "ALLOWED_ORIGINS=https://${FRONTEND_FQDN}"

6. Print deployment summary:
   - Frontend URL
   - Backend URL
   - Backend health check command
