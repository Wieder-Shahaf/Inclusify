# Adapter Versioning

## Directory Structure

```
ml/adapters/
  active.json              ← pointer to currently deployed adapter
  VERSIONING.md            ← this file
  qwen_r8_d0.2/            ← adapter directory (name = version ID)
    adapter_config.json    ← PEFT config (auto-generated by training)
    adapter_model.safetensors
    version.json           ← versioning metadata (accuracy, date, etc.)
    ...
  qwen_r8_d0.1/            ← previous version (kept for rollback)
    ...
```

## Versioning Scheme

Adapter directories are named: `{base_model}_{lora_params}`

- `qwen` — base model family
- `r8` — LoRA rank
- `d0.2` — dropout rate

Numeric version (in `version.json`): `{major}.{minor}.{patch}`
- Major: breaking change (new base model, task redefinition)
- Minor: new training data or significant accuracy improvement
- Patch: bug fix or minor tuning

## Adding a New Adapter Version

1. Train the new adapter (outputs to a new directory)
2. Add `version.json` with metadata (accuracy, training date, etc.)
3. Run the switch script to test locally:
   ```bash
   python ml/scripts/switch_adapter.py --list
   python ml/scripts/switch_adapter.py --adapter qwen_r8_d0.3 --dry-run
   ```
4. Deploy to the Azure VM:
   ```bash
   # On the VM:
   cd ~/inclusify && git pull
   python ml/scripts/switch_adapter.py --adapter qwen_r8_d0.3 --restart-service
   ```
5. Verify:
   ```bash
   curl http://localhost:8001/v1/models
   ```

## Rollback Procedure

If the new adapter behaves incorrectly, rollback by switching back to the previous version:

```bash
# Check what was previously active
cat ml/adapters/active.json

# Switch back (on the VM)
python ml/scripts/switch_adapter.py --adapter qwen_r8_d0.2 --restart-service

# Verify
curl http://localhost:8001/v1/models
```

Old adapter directories are **never deleted** — they remain available for rollback.
